Overview

Gref - Implementation of xref in golang for The Economist.

This service provides a cross reference API for the proposed ESB.

SETUP

  1. Clone this repository into your GOPATH.
  2. We use godep for dependency management - run godep restore to get the packages needed by this project.
  3. Run make build.
  4. Run build/gref and hit the endpoints.

MAKEFILE

The Makefile provides the following:

  • build: build the application binary and drop it in the build folder.
  • gomkbuild: build the application binary.
  • gomkxbuild: build all cross-platform binaries, using gox.
  • gomkclean: clean the project directory.
  • vet: run go tool vet on each source file.
  • lint: run golint on each source file.
  • fmt: run go fmt on the entire project.
  • test: run go test for all packages in the project.
  • race: run go test with race detection in all packages in the project.
  • cover: run tests with coverage report in all pkgs in the project.
  • printvars: print all variables defined in the Makefile.

API DOCUMENTATION

The API is specified using RAML and prettified using raml2html

GRef - Go Cross Reference is a service that provides mappings between application identifiers and a canonical identifier to aid in application integration.

The cross-referencing functionality enables the association of identifiers for equivalent entities created in different applications. For example, you can use cross references to associate a customer entity created in one application (with native id cust100) with an entity for the same customer in another application (with native id 145243322).

Cross-referencing facilitates the mapping of native keys for entities across applications. For example, correlate the same order across different ERP systems.

General Principles

  • Every object instance is associated with a canonical identifier, which is implemented as a UUID
  • If an application has knowledge fo the object instance then the application native identifier is mapped to the canonical
  • If an application does not have knowledge of the object instance then the mapping is NIL
  • The canonical identifier should be used for all API invocations

The easiest way to visualise these references is through tables, where each object type has a single table with each interested application having a column.

In the example above, we manage 3 instances of a Customer object that all have a canonical identifier. The first instance is known by Applications 1 and 3, the second instance is known by Application 2 and the third instance is known by Applications 2 and 3.

Functional Overview

GRef supports the following high-level functionality

  • Create and return a canonical identifier when sent an application identifier
  • Return the canonical identifier for an appliction identifier
  • Return all of the applictions associated with a canonical identifier
